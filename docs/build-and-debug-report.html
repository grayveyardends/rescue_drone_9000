Five processes, three protocols. The key idea: ArduPilot doesn't know it's in a simulator. The same firmware that runs on your Pixhawk runs here as a Linux binary (arducopter), and Gazebo plays the role of the physical world — it computes physics and feeds fake sensor data back. Everything above SITL (MAVProxy, mavros, the dashboard) works identically against your real drone; you just change one URL.

3 · The Dockerfile, line by line

Read this with the file open. Order matters enormously: Docker caches each step, and a step only re-runs if it or anything above it changed. That caching is what saved us — the build failed twice on network errors, and each retry resumed instead of restarting.

FROM ros:humble-ros-base-jammy
Start from the official ROS 2 Humble image on Ubuntu 22.04 ("jammy"). ros-base = ROS without GUI tools, ~1 GB smaller than desktop. This one decision fixes the Ubuntu version, Python version (3.10), and Gazebo version (11) for everything below.
SHELL ["/bin/bash", "-c"]
Default shell is sh, but ROS's setup.bash needs bash. Every later RUN uses bash because of this.
RUN apt-get update && apt-get install -y \ gazebo libgazebo-dev \ ros-humble-gazebo-ros-pkgs \ ros-humble-mavros ... \ && rm -rf /var/lib/apt/lists/*
One big apt layer. update and install must share a layer or you get stale-index bugs. The rm at the end deletes the package index (~50 MB) from the layer — a classic image-size habit. mavros is the ROS↔MAVLink translator; rosbridge-server exposes ROS over WebSocket for the dashboard; rmw-cyclonedds-cpp is the DDS middleware we pin later.
RUN /opt/ros/humble/lib/mavros/install_geographiclib_datasets.sh
mavros converts between GPS ellipsoid heights and mean-sea-level; it hard-crashes at startup without these geoid datasets. Everyone forgets this line once.
RUN git clone --depth 1 --recurse-submodules \ --shallow-submodules \ https://github.com/ArduPilot/ardupilot.git
--depth 1 = only the latest commit, not 15 years of history (saves ~2 GB). ArduPilot vendors dozens of libraries as git submodules; the build fails cryptically if you forget --recurse-submodules.
RUN pip3 install --no-cache-dir --retries 15 \ --timeout 120 pymavlink MAVProxy \ empy==3.3.4 ...
The --retries/--timeout flags were added after bug #1. empy==3.3.4 is pinned because ArduPilot's code generator breaks on empy 4.x — a famous ecosystem landmine. MAVProxy becomes our MAVLink router at runtime.
RUN cd /ardupilot && \ ./waf configure --board sitl && ./waf copter
ArduPilot uses waf, not make/cmake. --board sitl compiles the flight firmware as a normal x86 Linux program instead of ARM microcontroller code. This is the slowest step (~20 min) — deliberately placed after the rarely-changing layers, before the frequently-changing ones.
ENV PATH=$PATH:/ardupilot/Tools/autotest:...
Puts sim_vehicle.py (ArduPilot's SITL launcher) on the PATH so compose can just say sim_vehicle.py.
RUN for i in 1 2 3 4 5 6; do git clone ... \ ardupilot_gazebo && break; ...; done \ && cmake .. && make && make install
The C++ Gazebo plugin (libArduPilotPlugin.so) that shuttles motor commands and sensor data between Gazebo and SITL over UDP 9002/9003. The ugly retry loop is bug #2's fix — your network dropped mid-clone.
ENV GAZEBO_MODEL_PATH=... GAZEBO_RESOURCE_PATH=...
Where Gazebo searches for model://iris_sar style URIs — the plugin's models plus your sim/models.
COPY ros2_ws /sar/ros2_ws RUN ... colcon build
colcon is ROS 2's build tool; it generates the install/setup.bash you source to make ros2 launch sar_drone ... resolvable. This build gets shadowed at runtime — see bug #3.
ENV RMW_IMPLEMENTATION=rmw_cyclonedds_cpp ENTRYPOINT ["/entrypoint.sh"]
Pins the DDS vendor (ROS 2's pluggable network layer) so every container behaves identically, and routes all startup through the entrypoint script, which sources ROS, rebuilds the workspace if needed, kills the dead model database (bug #6), and sets ROS_LOCALHOST_ONLY=1 (bug #5).

And the compose file's one big idea: one image, four services. sitl, gazebo, ros, and dashboard all run sar_drone:latest with different command: lines, all with network_mode: host so that "connect to 127.0.0.1:14550" works between them without any port-mapping ceremony.

4 · The debugging log — nine failures, in order

This is the part you actually missed, and honestly the most valuable. Notice the pattern in every entry: observe the real symptom → find the first point in the chain where reality diverges from expectation → fix that, not the symptom.

#1 — The build that "succeeded" but didn't

Symptom
Background build exits with code 0, but docker images shows no image, and docker run says the image doesn't exist.
Debugging
Trusting the exit code was the mistake. Reading the actual log tail showed a pip stack trace ending in ReadTimeoutError: files.pythonhosted.org — Read timed out. The build had died at the pip step. So why exit 0? The command was docker build ... | tail -40 — in a pipeline, the shell reports the exit code of the last command. tail succeeded, so the pipeline "succeeded."
Fix
Added --retries 15 --timeout 120 to pip, and stopped piping build output (write to a log file instead, so the exit code is docker's own).

A pipeline's exit code is the last command's, not the interesting one's. In bash use ${PIPESTATUS[0]}, in zsh ${pipestatus[1]} — or don't pipe commands whose exit code you care about. And never claim success you haven't verified with your own eyes.

#2 — Same story, different downloader

Symptom
Second build attempt: fatal: unable to access 'github.com/.../ardupilot_gazebo': SSL connection timeout.
Debugging
Nothing to diagnose — but worth noticing that ArduCopter had already compiled successfully in a cached layer, so the retry resumed 20 minutes into the build, not from zero. This is why Dockerfile step ordering matters.
Fix
Wrapped the clone in a 6-attempt retry loop with sleeps. Networks flake; scripts that touch the network should assume it.

Design builds so that expensive steps cache and flaky steps retry. Your internet connection is part of your build system whether you like it or not.

#3 — The container that forgot its own package

Symptom
The ros container restart-loops with Package 'sar_drone' not found — even though the Dockerfile built it and the smoke test inside the raw image found it fine.
Debugging
The contradiction is the clue: the image has it, the running container doesn't. What differs between them? The compose file adds volumes: ./:/sar. That bind mount lays your host repo over /sar, hiding the ros2_ws/install/ directory that colcon created inside the image — your host copy had never been built.
Fix
The entrypoint now checks for install/setup.bash and runs colcon build --symlink-install on first start. Bonus: --symlink-install means Python edits on your host take effect on node restart, no rebuild.

A bind mount always wins over image content at that path. "Works in the image, breaks in compose" almost always means a mount is shadowing something.

#4 — MAVROS shouting into the void

Symptom
mavros logs endless VER: autopilot version service timeout; no heartbeat; dashboard shows link down.
Debugging
mavros listens on udp 14550. Who is supposed to send there? The sim_vehicle.py flags said --out=udp:...:14550 --no-mavproxy. Reading what those flags actually do revealed the contradiction: --out options are instructions for MAVProxy — and --no-mavproxy had disabled it. Nobody was forwarding anything. SITL itself only offers a TCP server on port 5760, single-client.
Fix
Run MAVProxy headless: -m "--daemon --non-interactive". It connects to SITL's tcp 5760 and fans out to udp 14550 (mavros) and 14551 (dashboard) — one flight controller, many listeners.

When A can't hear B, don't debug A — trace the whole route and find the hop that doesn't exist. Also: MAVProxy isn't optional decoration, it's the router.

#5 — ROS breaks when the wifi blinks

Symptom
Node logs full of ddsi_udp_conn_write to udp/10.73.x.x failed; topics and services flicker in and out.
Debugging
Those 10.x addresses are your wifi/VPN interfaces. ROS 2's transport, DDS, auto-discovers peers by multicasting on every network interface — and with network_mode: host, the containers see them all. When an interface flapped (your network was dropping all evening — same root cause as bugs #1 and #2), DDS kept trying to use its corpse.
Fix
First tried a hand-written CycloneDDS XML config pinning the loopback interface — it half-worked (topics yes, some services no). Replaced it with the built-in, battle-tested switch: ROS_LOCALHOST_ONLY=1. Everything runs on one machine, so DDS never needs to leave loopback.

Prefer a platform's built-in switch over your own clever config for the same goal. And know your transport: "ROS is magic pub/sub" ends the day discovery breaks.

#6 — Gazebo hangs calling a server that died years ago

Symptom
gzserver loads the world, prints Getting models from http://models.gazebosim.org/ … and hangs forever. No error, no world.
Debugging
The world file said <include><uri>model://sun</uri>. Any model not found locally makes Gazebo Classic query its online database — which was shut down years ago; the domain just blackholes. Then a second layer: after fixing sun/ground, the iris model itself failed with Couldn't Find Child Link [gimbal_small_2d::base_link] — the stock iris_with_ardupilot includes a camera gimbal that also only exists in the dead database.
Fix
Inlined the sun (a <light>) and ground plane directly in the world file; created sim/models/iris_sar — the iris minus the gimbal include; set GAZEBO_MODEL_DATABASE_URI="" so nothing can ever try the network again. Side benefit: the sim is now fully offline-capable, which you wanted anyway.

A hang is a timeout you haven't waited out yet — go find the network call. Old ecosystems accumulate dead infrastructure; vendoring your dependencies is self-defense.

#7 — Wrong message type on /rosout

Symptom
rosbridge error: Tried to register topic /rosout with type std_msgs/String but it is already established with type rcl_interfaces/msg/Log.
Debugging
Self-inflicted and honest: the dashboard relay subscribed to every topic as std_msgs/String for convenience. /rosout (ROS's logging firehose) is a structured Log message. ROS topics are strongly typed; you can't just declare a type and hope.
Fix
Subscribe with the correct type and format [node] message from its fields.

Check a topic's real type with ros2 topic info /rosout before subscribing. Typed interfaces fail loudly at the boundary — that's a feature.

#8 — The service that exists but can't be found

Symptom
Takeoff does nothing. mission_bridge logs service /mavros/cmd/arming unavailable — yet ros2 service list plainly shows the service, and its neighbor /mavros/set_mode works.
Debugging
Narrowed it with a manual ros2 service call — same result: hangs on "waiting for service." So it's not our node's fault; the mavros command plugin's services are advertised in the graph but never become callable — a known rough edge of mavros2 service discovery (mavros even fails to call its own command plugin at startup: that early VER: command plugin service call failed message was a clue hiding in plain sight). Rather than fight an upstream bug, ask: who else can arm a drone? Any ground station — by sending a plain MAVLink COMMAND_LONG. And the dashboard already holds a direct MAVLink connection for telemetry.
Fix
Split responsibilities: arm/disarm/mode/takeoff/land/RTL go out the dashboard's pymavlink link (the exact bytes Mission Planner would send); mission_bridge keeps goto/search patterns via topic-published setpoints, which work perfectly. Bonus resilience: basic flight control now survives the entire ROS stack dying.

When one layer is unreliable, ask what the layer below it speaks — mavros services are just a wrapper around MAVLink commands you can send yourself. Re-reading early "harmless" log errors often pays off later.

#9 — PreArm: Main loop slow

Symptom
Everything connected, yet arming always refused: PreArm: Main loop slow (272Hz < 400Hz) and a gyro-rate complaint.
Debugging
This one is physics, not plumbing. Gazebo and SITL run in lockstep: each 2.5 ms physics step feeds one sensor sample to ArduPilot, whose control loop expects 400 steps per real second. Your machine sustains ~272 physics steps/sec, so the firmware's scheduler measures itself running slow — and on a real drone a slow control loop crashes aircraft, so PreArm refuses. It's a safety check doing its job against a simulator that can't keep up.
Fix
SCHED_LOOP_RATE 200 in sim/params/sar_sitl.parm, loaded via --add-param-file. At 200 Hz the machine has headroom, checks pass, and the copter flies fine. (Alternative rejected: disabling arming checks entirely — never train yourself to bypass safety systems.)

Simulators borrow your CPU as their physics engine; "the drone won't arm" can literally mean "your laptop is too slow." Fix the parameter that encodes the assumption, don't disable the check that caught it.

After #9: takeoff commanded through the dashboard API → alt = 10.0 m, GUIDED, armed confirmed over the telemetry WebSocket, then LAND → DISARMED. That closing loop — verify through the same interface the user will use — is the habit worth stealing.

5 · How a click becomes a flying drone

Trace one command end-to-end; if you can narrate this chain from memory, you understand the system.

  1. You click the map. The browser JS builds {"cmd":"goto","lat":9.93,"lon":76.26,"alt":25} and POSTs it to /api/command.
  2. FastAPI publishes it as a JSON string on the ROS topic /sar/command, via its roslibpy connection to rosbridge (WebSocket, port 9090).
  3. Two subscribers hear it. mission_bridge handles goto: it publishes a GeoPoseStamped to /mavros/setpoint_position/global. (Had it been takeoff, the dashboard's own MAVLink executor would have taken it instead — bug #8's split.)
  4. mavros translates that ROS message into a MAVLink SET_POSITION_TARGET_GLOBAL_INT packet and sends it to udp 14550.
  5. MAVProxy routes it to SITL's TCP serial port — exactly as a telemetry radio would deliver it to your Pixhawk.
  6. ArduPilot's guidance logic accepts the target (GUIDED mode), runs its position → velocity → attitude → motor cascade, and outputs four PWM values.
  7. The Gazebo plugin receives those PWM values over udp 9002, spins the rotor joints, the LiftDrag plugin turns rotation into thrust, physics integrates, and the pose + IMU + GPS go back over 9003.
  8. Telemetry returns up the same chain; the dashboard's pymavlink thread parses GLOBAL_POSITION_INT and pushes JSON over the WebSocket; the marker on your map moves.

The LLM path is the same chain with a different head: image → /sar/analyze_b64 → llm_agent → llama-server → structured <<action_start>> text → regex parse → the same /sar/command topic your buttons use. The model is just another operator at the same console — which is also the safety argument: everything it can do, you can see and veto.

6 · ROS vs MAVLink, finally clear

MAVLinkROS 2
What it isA compact binary wire protocol for talking to a flight controllerA middleware for composing robot software from many programs
Vocabulary~300 fixed message types (HEARTBEAT, COMMAND_LONG, GLOBAL_POSITION_INT…)Unlimited user-defined typed topics/services/actions
TransportAny byte pipe: serial radio, UDP, TCPDDS over UDP with automatic peer discovery
Speaks itPixhawk, ArduPilot, Mission Planner, QGroundControl, your dashboard's pymavlinkmavros, rosbridge, your two nodes, rviz, rosbag
In this projectEverything touching the autopilotEverything touching your application logic

mavros is the adapter between the two worlds — and bug #8 taught the real lesson: adapters leak. Knowing the layer beneath your abstraction (being able to send raw COMMAND_LONG yourself) is what turned an upstream bug from a wall into a detour.

7 · Exercises: make it yours

Ordered easy → hard. Each touches a layer you watched get built; break things freely — git checkout and docker compose down undo everything.

  1. Watch the nervous system. docker exec -it drone_rescue-ros-1 bash, source ROS, then ros2 topic list, ros2 topic echo /mavros/state, ros2 topic hz /mavros/global_position/global. Publish a command by hand with ros2 topic pub and watch the drone obey.
  2. Speak raw MAVLink. Ten-line Python: mavutil.mavlink_connection('udp:127.0.0.1:14551'), wait for heartbeat, print message types as they arrive. You'll never see the telemetry panel as magic again.
  3. Move the survivors. Edit sar_world.world — reposition the rubble, add a third survivor, change the sun angle. Restart only the gazebo service.
  4. Add a dashboard panel. A "flight timer" or message-rate counter: one div in index.html, a few lines in the WebSocket handler. The panel drag system will pick it up automatically.
  5. Add a command. Implement {"cmd":"orbit", ...} end-to-end: a button in the UI, a cmd_orbit method in mission_bridge publishing a circle of setpoints. This walks you through the whole chain in section 5.
  6. Write your own script in scripts/ and make the LLM call it by adding it to the system prompt's action list.
  7. Fly the F550. Swap model://iris_sar for model://f550 in the world, set FRAME_CLASS 2 in the param file, and tune until it hovers. This is real integration work — expect it to fight back like bugs #6–#9 did.
  8. Break it on purpose. Re-add --no-mavproxy and watch bug #4 happen to you; set SCHED_LOOP_RATE back to 400 and meet bug #9. Reproducing a bug you've read about is how the diagnosis becomes yours.
  9. Record and replay. Record a flight with the Bags tab, then ros2 bag play it back and watch the dashboard replay history — understand why bags are the black box recorder of robotics.
  10. Point it at the real drone. When ready: FCU_URL=serial:///dev/ttyACM0:57600, props off, on a bench. Everything you've learned transfers unchanged — that was the whole point of the architecture.
